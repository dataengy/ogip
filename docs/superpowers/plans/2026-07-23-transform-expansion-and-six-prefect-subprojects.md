# Transform expansion, ODTS-synced SQLMesh DQ, and six separated Prefect sub-projects — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** (1) Extend the transform SSoT so **every raw source flows through staging into core/fs** across all engines and ODTS fixtures; (2) declare **comprehensive DQ once in the ODTS spec** and make the SQLMesh adapter *project it into audits* (today it silently drops every check); (3) restructure the Prefect orchestration into **six separated, individually-deployable sub-projects** over a shared step library.

**Architecture:** `spec/sql/` (the `@bruin`-format ODTS authoring surface, ADR-0016) is the single source of truth for models *and* their DQ. Adapters (`src/ogip/spec_compile/to_*.py`) project it to SQLMesh / dbt / Bruin. Three drifts exist today and this plan closes them: (a) core/fs consume only the rawg spine while four sources dead-end in staging; (b) `to_sqlmesh._model_text` emits `MODEL(name, kind)` and **discards `columns.checks`**; (c) all six orchestration setups are modules sharing one `_common.py`, not separated deployables. The unifying principle stays intact: **declare once in the spec, project everywhere; the registry/spec is SSoT, no engine re-implements a model or a check.**

**Tech Stack:** Python 3.13 (pyright strict, ruff), `@bruin`-portable SQL, SQLMesh (built-in + custom audits), dbt (tests), Bruin (checks), Prefect 3 (`@flow`/`@materialize`, `prefect.yaml` deployments), Dagster 1.13.x (native `dagster_dlt`/`dagster_dbt`, run *under* Prefect), DuckDB warehouse, pytest.

## Global Constraints

- **Spec is SSoT for DQ.** Every correctness check is authored in `spec/sql` `columns.checks:` / `checks:` (ODTS §5–6 vocabulary) and projected by the adapters. No audit is hand-written in `transform/sqlmesh/models/` — those files are *generated*.
- **ODTS §6 boundary — checks ≠ monitors.** `checks:` hold correctness constraints only (`not_null`, `unique`, `non_negative`, `between(a,b)`, `accepted_values`, `fk`). Freshness / row-count **monitors** MUST NOT enter `checks:`; they belong to the ODOS `monitoring.yml` / `dq/` layer. "Comprehensive DQ" is split accordingly — correctness → SQLMesh audits (synced from ODTS); monitors → ODOS/dq.
- **ODTS fixtures track reality.** `spec/ODTS/examples/` is a conformance subset of `spec/sql/`; any model or check added to `spec/sql` is mirrored into the ODTS examples in the same commit (the sync directive).
- **Generated dirs are outputs, never hand-edited.** `transform/{sqlmesh,dbt,bruin}/models|assets/` are regenerated by the adapters. Editing them by hand is a defect; the fix goes in `spec/sql` + the adapter.
- **No cross-source key is assumed.** The five sources share no natural key; integration into core goes through an explicit title-normalization bridge, and join coverage is itself a DQ check — never a silent inner join that drops rows.
- **Layer-0 law holds.** Raw stays 1:1 AS-IS; all casting/normalization is staging's job. Scrapers stay `publishable: false` and demo-safe by default.
- **Registry is SSoT for tasks; `_shared` is SSoT for steps.** The six sub-projects import shared step functions; none forks a model, a task body, or a step.
- **Commit binding.** Every commit carries `Refs: #37` (ODOS/ODTS wiring) and, where source-specific, `Refs: #18` / `Refs: #19`.
- **PROPOSE-ONLY for skills/agents/hooks.** Proposed here; created via `/create-skill` / `/upsert-skill` / `/sync-project-agents` after approval. Scripts under `~/.ai/skills/_scripts/` are the preferred deterministic surface.

## Scope note (decomposition)

Three independent, sequenced subsystems, each shippable on its own — implement in order because each builds on the prior:
1. **Part 1 — Full raw lineage** (models). Foundation.
2. **Part 2 — ODTS-synced SQLMesh DQ** (checks → audits). Builds on Part 1's models.
3. **Part 3 — Six separated Prefect sub-projects** (orchestration). Wraps the completed pipeline.

Parts 4 (scripts/skills/hooks) and 5 (persistence) are cross-cutting and land incrementally with each part.

```sql
/* @bruin
name: staging.stg_game_match
type: duckdb.sql
materialization: {type: view}
owner: data-eng@ogip
tags: [staging, bridge, daily]
depends: [staging.stg_games, staging.stg_metacritic_games, staging.stg_opencritic_games, staging.stg_psn_concepts, staging.stg_steamcharts_apps]
columns:
  - name: game_sk
    checks: [{name: not_null}]
  - name: match_key
    checks: [{name: not_null}]
@bruin */
with spine as (
    select md5(cast(game_id as varchar)) as game_sk,
           regexp_replace(lower(name), '[^a-z0-9]', '', 'g') as match_key
    from staging.stg_games
)
select game_sk, match_key from spine
```

## Part 2 — ODTS-synced comprehensive DQ, projected into SQLMesh audits

**Current state:** `to_sqlmesh._model_text` emits `MODEL(name, kind)` + SQL and **drops `columns.checks` entirely**. `Asset.meta` retains the full header (columns + checks), so the fix is localized to the adapter — no parser change. Existing checks: `core.game`/`fs` only; raw/staging carry none.

**Design:** enrich checks in `spec/sql` (ODTS §5–6 vocabulary), then teach `to_sqlmesh` to render them as SQLMesh audits. Monitors (freshness/rowcount) are explicitly OUT of `checks:` (ODTS §6) and are added as clearly-labelled SQLMesh-native audits under `transform/sqlmesh/audits/` that are NOT spec-synced (documented as engine extras), plus their real home in ODOS `monitoring.yml`.

### Task 2.1: `to_sqlmesh` renders `columns.checks` as built-in audits

**Files:** Modify `src/ogip/spec_compile/to_sqlmesh.py`; Test `src/tests/unit/test_to_sqlmesh_audits.py`

**Interfaces:**
- Produces: `_audits(asset: Asset) -> str` — an `audits (...)` clause for the MODEL block. - [ ] **Step 3:** Implement `_audits` + splice it into `_model_text`'s `MODEL(...)` block:

```python
def _audit_for(col: str, chk: dict[str, object]) -> str | None:
    name = chk["name"]
    args = chk.get("args") or []
    if name == "not_null":
        return f"not_null(columns := ({col}))"
    if name == "unique":
        return f"unique_values(columns := ({col}))"
    if name == "non_negative":
        return f"accepted_range(column := {col}, min_v := 0)"
    if name == "between" and len(args) == 2:
        return f"accepted_range(column := {col}, min_v := {args[0]}, max_v := {args[1]})"
    if name == "accepted_values" and args:
        vals = ", ".join(f"'{v}'" for v in args)
        return f"accepted_values(column := {col}, is_in := ({vals}))"
    return None  # unknown → surfaced by the vocabulary gate (Task 2.3), never silently dropped


def _audits(asset: Asset) -> list[str]:
    out: list[str] = []
    for col in asset.meta.get("columns") or []:
        for chk in col.get("checks") or []:
            rendered = _audit_for(col["name"], chk)
            if rendered:
                out.append(rendered)
    return out
```

- [ ] **Step 4:** In `_model_text`, when `_audits(asset)` is non-empty, extend the header: `MODEL (\n  name {name},\n  kind {kind},\n  audits (\n    {",\n    ".join(audits)}\n  )\n);`.

## Part 3 — Six separated, individually-deployable Prefect sub-projects

**Current state:** `pipelines/flows/engines/prefect_{sqlmesh,dbt,bruin,dagster,opendbt,sqlmesh_dbt}.py`, all one-liners over `make_engine_flow` in a shared `_common.py`; no `prefect.yaml`, no `deployments/`. Decision (locked): **project-level separation, shared steps** — each engine becomes its own sub-project directory with `prefect.yaml` + `flow.py` + `deployments/`, importing steps from a shared library. Ingestion tooling (dlt/ingestr/airbyte/scraping) is available in every sub-project — shared by default, custom where a tool integrates specially (dlt-as-`dagster_dlt` in the dagster project; dlt-as-opendbt-source in opendbt). In the dagster sub-project **Prefect is the outer orchestrator and Dagster runs under it** (`dg launch`), per `/call-dagster-from-prefect`.

```
pipelines/
```
